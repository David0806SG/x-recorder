01 · The brief

A Clipchamp-style recorder, in the browser

  • Record the screen and capture the speaker via the laptop camera
  • Camera shown as a movable picture-in-picture bubble, burned into the final video
  • Mix microphone + system audio; countdown, pause/resume, preview & download
  • Self-imposed constraints: vanilla JS, no build step, no dependencies — runs as static files

The reference was Clipchamp's recorder. Constraint: keep it dependency-free.

The brief was simple: a screen recorder like Clipchamp's, that also captures you on webcam as a little bubble over the screen — and bakes both into one video file.

I deliberately constrained it to plain HTML, CSS and JavaScript — no framework, no build step, no dependencies. Everything uses native browser APIs.

03 · Architecture

How it works under the hood

Capture & compositing

  • Canvas compositing — screen + camera drawn onto one <canvas> that is both the live preview and the recorded surface (WYSIWYG)
  • Worker-driven draw loop — frames keep flowing even when the tab is in the background

Recording & audio

  • MediaRecorder → VP9/Opus WebM, codec-negotiated
  • Web Audio mixing — mic + system audio with live mute gains
9 ES modules · 0 dependencies — plain HTML, CSS and JavaScript, served as static files.

The canvas-is-the-recording trick is the core idea. Worker loop is the key robustness trick.

Architecturally, the clever bit is that a single canvas is both what you see and what gets recorded — so the camera bubble is always exactly where you placed it.

The draw loop is driven by a Web Worker timer instead of requestAnimationFrame, because rAF freezes when you switch tabs — which is exactly when you're recording another tab. Audio is mic plus system sound mixed through the Web Audio API. It's nine small modules, zero dependencies.

04 · The app

Export, tuned for file size

In-browser MP4 export — choose resolution (1080 / 720 / 480p) and frame rate (30 / 24 / 15 fps) to trade quality for size. The low end comes out ~71% smaller.

Highlight that this is real in-browser transcoding, no server.

When you stop, you preview the recording and export it to MP4 — choosing both resolution and frame rate to control file size. Picking the low settings produces a file about seventy percent smaller. And this transcoding happens entirely in the browser — no server, no FFmpeg.

05 · Engineering wins

The problems worth solving

  • Background-tab recording — a Web Worker tick replaces requestAnimationFrame so recording never freezes when you switch tabs
  • QuickTime-compatible MP4 — H.264 packaging fixed (avc3 → avc1) and verified against Apple's own AVFoundation
  • True in-browser transcode — WebM re-encoded to MP4 via a second MediaRecorder, no server
  • WYSIWYG output — bubble position, size and shape changes are burned into the video live

These are the non-obvious wins — the QuickTime fix is a good story.

A few problems were genuinely worth solving. Keeping the recording alive when the tab is backgrounded. Making the MP4 actually open in QuickTime — that was a codec-packaging fix, validated against Apple's own media framework. Doing the WebM-to-MP4 transcode purely client-side. And making the output truly what-you-see-is-what-you-get.
